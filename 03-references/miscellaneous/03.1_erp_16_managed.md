# ERPNext v16 — Managed Infrastructure Setup Guide

## Purpose

This guide covers setting up ERPNext v16 where infrastructure components are split across managed services and dedicated VMs:

- **ERP App Server** — dedicated VM (EC2 on AWS / Compute Engine on GCP)
- **Database (MariaDB)** — dedicated VM in same VPC (not managed — GCP/AWS have no managed MariaDB)
- **Redis (Cache + Queue)** — managed service (ElastiCache on AWS / Memorystore on GCP)

---

## Why Managed Redis but Not Managed Database

| Service | AWS | GCP | Why |
|---|---|---|---|
| Redis | ElastiCache ✔ | Memorystore ✔ | Both offer managed Redis |
| MariaDB | RDS MariaDB ✔ | ❌ Not available | GCP has no managed MariaDB |
| MySQL | RDS MySQL ✔ | Cloud SQL MySQL ✔ | MySQL 8.4 incompatible with Frappe |

GCP only offers managed MySQL — MySQL 8.4 is incompatible with Frappe v16 (MyISAM disabled, `mysql_native_password` removed). Therefore: **MariaDB on a dedicated VM** is the correct path for GCP. AWS users can use RDS MariaDB.

## Section 1 — AWS Setup

### 1.1 — AWS Architecture

```
VPC: 10.0.0.0/16
  ├── Public Subnet (10.0.1.0/24)
  │     └── ERP App EC2 (t3.large)
  │           Elastic IP assigned
  │
  ├── Private Subnet (10.0.2.0/24)
  │     └── MariaDB EC2 (t3.medium)
  │           No public IP
  │
  └── Private Subnet (10.0.3.0/24)
        └── ElastiCache Redis (cache.t3.micro / cache.r6g.large)
              No public IP
```

### 1.2 — Provision MariaDB EC2 (AWS)

```bash
# Launch EC2 via AWS CLI
aws ec2 run-instances \
  --image-id ami-0c7217cdde317cfec \
  --instance-type t3.medium \
  --subnet-id <private-subnet-id> \
  --security-group-ids <mariadb-sg-id> \
  --no-associate-public-ip-address \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=erp-mariadb-vm}]'
```

Security Group for MariaDB EC2:
```
Inbound:
  Port 3306  TCP  Source: ERP App VM security group
  Port 22    TCP  Source: bastion/your IP (for SSH)
Outbound:
  All traffic
```

### 1.3 — Provision ElastiCache Redis (AWS)

```bash
# Create Redis subnet group first
aws elasticache create-cache-subnet-group \
  --cache-subnet-group-name erp-redis-subnet \
  --cache-subnet-group-description "ERP Redis Subnet" \
  --subnet-ids <private-subnet-id-1> <private-subnet-id-2>

# Create Redis cluster
aws elasticache create-cache-cluster \
  --cache-cluster-id erp-redis \
  --cache-node-type cache.t3.micro \
  --engine redis \
  --num-cache-nodes 1 \
  --cache-subnet-group-name erp-redis-subnet \
  --security-group-ids <redis-sg-id>
```

Security Group for ElastiCache:
```
Inbound:
  Port 6379  TCP  Source: ERP App VM security group
Outbound:
  All traffic
```

Get Redis endpoint:
```bash
aws elasticache describe-cache-clusters \
  --cache-cluster-id erp-redis \
  --show-cache-node-info \
  --query 'CacheClusters[0].CacheNodes[0].Endpoint'
```

```
VPC: erp-vpc
```

```bash
# Create VPC
gcloud compute networks create erp-vpc \
  --subnet-mode=custom

# App subnet
gcloud compute networks subnets create erp-app-subnet \
  --network=erp-vpc \
  --region=asia-south1 \
  --range=10.0.1.0/24

# DB subnet
gcloud compute networks subnets create erp-db-subnet \
  --network=erp-vpc \
  --region=asia-south1 \
  --range=10.0.2.0/24
```

### 2.3 — Provision MariaDB VM (GCP)

```bash
gcloud compute instances create erp-mariadb-vm \
  --zone=asia-south1-a \
  --machine-type=e2-standard-2 \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --network=erp-vpc \
  --subnet=erp-db-subnet \
  --no-address \
  --boot-disk-size=50GB \
  --tags=mariadb-server
```

Firewall rule — allow app VM to reach MariaDB:
```bash
gcloud compute firewall-rules create allow-mariadb-from-app \
  --network=erp-vpc \
```

### 2.5 — Provision ERP App VM (GCP)

```bash
gcloud compute instances create erp-app-vm \
  --zone=asia-south1-a \
  --machine-type=e2-standard-2 \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --network=erp-vpc \

  --region=asia-south1
```

```bash
  --target-tags=erp-app

  --network=erp-vpc \
  --allow=tcp:22 \
```

## Section 3 — MariaDB and Redis VM Setup (Same for AWS and GCP)

SSH into the MariaDB VM and run:

### 3.1 — Install MariaDB

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y mariadb-server mariadb-client
sudo systemctl start mariadb
sudo systemctl enable mariadb
```

### 3.2 — Configure Remote Access

MariaDB binds to `127.0.0.1` by default — change to accept remote connections:

```bash
sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
```

Find and change:
```ini
# FROM:
bind-address = 127.0.0.1

# TO:
bind-address = 0.0.0.0
```

### 3.3 — Configure Charset

Create a new config file — do not edit `50-server.cnf` directly:

```bash
sudo nano /etc/mysql/mariadb.conf.d/99-frappe.cnf
```

> ⚠️ **Section order is critical — `[server] → [mysql] → [mysqld]`. Wrong order causes collation errors.**

```ini
[server]

[mysql]
default-character-set = utf8mb4

[mysqld]
character-set-client-handshake = FALSE
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
```

### 3.4 — Secure MariaDB and Switch Auth

```bash
sudo mysql
```

```sql
-- Switch root to password auth (prevents unix_socket Access Denied during bench new-site)
ALTER USER 'root'@'localhost'
  IDENTIFIED VIA mysql_native_password
  USING PASSWORD('<db-root-password>');

-- Create remote root user for bench (connects from app VM IP)
CREATE USER 'root'@'%'
  IDENTIFIED BY '<db-root-password>';
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' WITH GRANT OPTION;

-- Remove anonymous users and test DB
DELETE FROM mysql.user WHERE User='';
DROP DATABASE IF EXISTS test;
FLUSH PRIVILEGES;
EXIT;
```

### 3.5 — Restart and Verify

```bash
sudo systemctl restart mariadb

# Verify charset
mysql -u root -p'<db-root-password>' \
  -e "SHOW VARIABLES LIKE 'character_set_server';"
# Must return: utf8mb4

# Verify remote root user exists
mysql -u root -p'<db-root-password>' \
  -e "SELECT user, host, plugin FROM mysql.user WHERE user='root';"
# Must show root@% with mysql_native_password
```

### 3.6 - Install Redis
 
Same as v15.
 
```bash
sudo apt install -y redis-server
sudo systemctl start redis-server
sudo systemctl enable redis-server
redis-cli ping   # Must return: PONG
```

Make sure that redis is listening on 0.0.0.0 instead of 127.0.0.1

```bash
sudo vi /etc/redis/redis.conf
```

Find and change:
```ini
# FROM:
bind-address = 127.0.0.1

# TO:
bind-address = 0.0.0.0
```

> ⚠️ **Do NOT install `mariadb-server` or `redis-server` service on the app VM.**
> Install only client tools and dev headers needed for bench to compile and connect.

```bash
sudo apt update && sudo apt upgrade -y

echo '* libraries/restart-without-asking boolean true' | sudo debconf-set-selections

sudo apt install -y \
  git curl wget \
  python3 python3-dev python3-pip python3-venv \
  build-essential \
  libffi-dev libssl-dev \
  software-properties-common \
  xvfb libfontconfig \
  cron \
  nginx \
  supervisor \
  pkg-config \
  mysql-client \
  libmariadb-dev \
  libmariadb-dev-compat \
  redis-tools

# Install redis-server binary ONLY (for bench init version detection)
# Stop and disable service immediately — managed Redis handles actual connections
sudo apt install -y redis-server
sudo systemctl stop redis-server
sudo systemctl disable redis-server

# Verify binary exists but service is dead
redis-server --version     # ✔ binary present for bench init
sudo systemctl status redis-server | grep Active
# Must show: inactive (dead)
```

> **Why `redis-server` package but service disabled?**
> `bench init` runs `redis-server --version` to detect Redis version for generating config file syntax. Exit code 127 (binary not found) causes `bench init` to fail. The service does not need to run — only the binary needs to exist.

## Section 5 — Pre-Connection Verification (CRITICAL)

> ⚠️ **Run ALL checks below before proceeding. Every single one must pass.**
> Skipping this step causes failures during `bench new-site` or `install-app` that are hard to diagnose.

### 5.1 — MariaDB Connectivity Check

```bash
# From the ERP App VM:

# 1. Basic connectivity
mysql -h <mariadb-vm-private-ip> -u root -p'<db-root-password>' \
  -e "SELECT 1 AS connected;"
# Must return: connected = 1
# If hangs → firewall blocking port 3306 between subnets

# 2. Auth plugin check
mysql -h <mariadb-vm-private-ip> -u root -p'<db-root-password>' \
  -e "SELECT user, host, plugin FROM mysql.user WHERE user='root';"
# root@% must show: mysql_native_password

# 3. Charset check
mysql -h <mariadb-vm-private-ip> -u root -p'<db-root-password>' \
  -e "SHOW VARIABLES LIKE 'character_set_server';"
# Must return: utf8mb4

# 4. Remote root can create databases
mysql -h <mariadb-vm-private-ip> -u root -p'<db-root-password>' \
  -e "CREATE DATABASE test_connectivity; DROP DATABASE test_connectivity;"
# Must complete without error
```

### 5.2 — Redis Connectivity Check

```bash
# From the ERP App VM:

# 1. Basic ping
redis-cli -h <redis-ip> -p 6379 ping
# Must return: PONG
# If hangs → firewall blocking port 6379

# 2. Write and read test
redis-cli -h <redis-ip> -p 6379 SET test_key "bench_connected"
redis-cli -h <redis-ip> -p 6379 GET test_key
# Must return: bench_connected

# 3. Cleanup
redis-cli -h <redis-ip> -p 6379 DEL test_key
```

### 5.4 — Final Pre-flight Checklist

Run this checklist before every major operation:

```bash
echo "=== MariaDB Check ==="; mysql -h <mariadb-ip> -u root -p'<db-root-password>' -e "SELECT 'MariaDB OK' AS status;" 2>/dev/null && echo "✔ MariaDB reachable" || echo "✘ MariaDB FAILED"

echo "=== Redis Check ==="; redis-cli -h <redis-ip> -p 6379 ping 2>/dev/null | grep -q PONG && echo "✔ Redis reachable" || echo "✘ Redis FAILED"

echo "=== Redis Config Check ==="; grep "redis_queue" ~/frappe-bench/sites/common_site_config.json | grep -v "127.0.0.1" && echo "✔ Redis config points to managed Redis" || echo "✘ Redis config still points to localhost"

echo "=== DB Host Check ==="
SITE_CONFIG=~/frappe-bench/sites/<client-application-name>/site_config.json
[ -f "$SITE_CONFIG" ] && grep "db_host" "$SITE_CONFIG" && echo "✔ db_host is set" || echo "⚠ site_config.json not yet created (pre-new-site)"
```

All four must show ✔ before proceeding.

## Section 7 — Production Setup
 
### 7.1 — bench setup production
 
```bash
sudo env PATH=$PATH:/home/frappe/.local/bin bench setup production frappe
```
 
### 7.1a — Build Frontend Assets (CRITICAL — do this before Nginx setup)
 
> ⚠️ **`bench setup production` does NOT build assets automatically.**
> Without this step Nginx returns 404 on all `/assets/` requests — site loads with no CSS, no JS, broken UI.
> In v16 assets use hashed filenames (`desk.bundle.34HFSR7T.css`) not flat names (`desk.css`).
 
```bash
cd ~/frappe-bench
bench build
 
# Verify assets exist on disk
ls ~/frappe-bench/sites/assets/frappe/dist/css/ | head -3
# Must show files like: desk.bundle.XXXXXXXX.css
```
 
> ⚠️ **Do NOT run the curl test here yet.**
> The curl test will return 403 Permission Denied until Step 7.4 (chmod) runs.
> Come back and run the curl test after Step 7.4 is complete:
> ```bash
> # Run this AFTER Step 7.4 (chmod permissions):
> curl -I http://localhost/assets/frappe/dist/css/desk.bundle.<hash>.css | head -2
> # Must show: HTTP/1.1 200 OK
> ```
 
### 7.2 — Supervisor Symlink
 
```bash
sudo ln -sf /home/frappe/frappe-bench/config/supervisor.conf \
  /etc/supervisor/conf.d/frappe-bench.conf
 
# Full daemon restart first — prevents CANT_REREAD conflict error
sudo systemctl restart supervisor
sleep 3
 
sudo supervisorctl reread
sudo supervisorctl update
```
 
> ⚠️ **If you see `CANT_REREAD: names unknown program frappe-bench-redis-cache`:**
> This happens when supervisord already has a stale config loaded in memory that conflicts with the new symlinked config. It is non-fatal if all processes show `RUNNING` in `supervisorctl status`.
> Clean fix — restart the supervisor daemon itself:
> ```bash
> sudo systemctl restart supervisor
> sleep 3
> sudo supervisorctl reread && sudo supervisorctl update
> sudo supervisorctl status
> ```
 
### 7.3 — Remove Local Redis From Supervisor
 
Since managed Redis handles all Redis operations — remove bench's local Redis process groups:
 
```bash
nano ~/frappe-bench/config/supervisor.conf
```
 
Comment out:
```ini
# [group:frappe-bench-redis]
# [program:frappe-bench-redis-cache]
# [program:frappe-bench-redis-queue]
```
 
```bash
sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl status
# Redis process groups must NOT appear in output
```
 
### 7.4 — Static Asset Permissions
 
```bash
chmod o+x /home/frappe
```

```nginx
        try_files $uri =404;
```

## Section 8 — Pre-Operation Checklist (Run Before Every Major Change)

Before any of these operations — `bench migrate`, `install-app`, `bench update`, site restore, production setup:

```bash
#!/bin/bash
# Save as: ~/check-erp-connections.sh
# Run: bash ~/check-erp-connections.sh

MARIADB_IP="<mariadb-vm-private-ip>"
MARIADB_PASS="<db-root-password>"
REDIS_IP="<redis-ip>"
REDIS_PORT="6379"
BENCH_PATH="/home/frappe/frappe-bench"
SITE_NAME="<client-application-name>"

echo ""
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "  ERP Pre-Operation Connection Check"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo ""

# 1. MariaDB
mysql -h "$MARIADB_IP" -u root -p"$MARIADB_PASS" \
  -e "SELECT 1;" > /dev/null 2>&1 \
  && echo "✔ MariaDB reachable at $MARIADB_IP:3306" \
  || echo "✘ MariaDB FAILED at $MARIADB_IP:3306 — check firewall/service"

# 2. MariaDB charset
CHARSET=$(mysql -h "$MARIADB_IP" -u root -p"$MARIADB_PASS" \
  -se "SHOW VARIABLES LIKE 'character_set_server';" 2>/dev/null | awk '{print $2}')
[[ "$CHARSET" == "utf8mb4" ]] \
  && echo "✔ MariaDB charset: utf8mb4" \
  || echo "✘ MariaDB charset: $CHARSET (must be utf8mb4)"

# 3. Redis
redis-cli -h "$REDIS_IP" -p "$REDIS_PORT" ping 2>/dev/null | grep -q PONG \
  && echo "✔ Redis reachable at $REDIS_IP:$REDIS_PORT" \
  || echo "✘ Redis FAILED at $REDIS_IP:$REDIS_PORT — check firewall/service"

# 4. Redis config in common_site_config
grep -q "$REDIS_IP" "$BENCH_PATH/sites/common_site_config.json" \
  && echo "✔ common_site_config.json points to managed Redis" \
  || echo "✘ common_site_config.json Redis mismatch — update redis_* entries"

# 5. Site db_host
SITE_CONFIG="$BENCH_PATH/sites/$SITE_NAME/site_config.json"
if [ -f "$SITE_CONFIG" ]; then
  grep -q "$MARIADB_IP" "$SITE_CONFIG" \
    && echo "✔ site_config.json db_host = $MARIADB_IP" \
    || echo "✘ site_config.json db_host mismatch — check db_host value"
else
  echo "⚠ site_config.json not found (pre-new-site — expected)"
fi

# 6. Supervisor processes
sudo supervisorctl status 2>/dev/null | grep -q RUNNING \
  && echo "✔ Supervisor processes running" \
  || echo "✘ No Supervisor processes running — check supervisorctl status"

echo ""
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
```

Make executable and run:
```bash
chmod +x ~/check-erp-connections.sh
bash ~/check-erp-connections.sh
```

All items must show ✔ before proceeding with any operation.

## Section 10 — AWS vs GCP Difference Summary

| Step | AWS | GCP |
|---|---|---|
| Managed Redis | ElastiCache | Memorystore |
| Database | RDS MariaDB (managed) or EC2 MariaDB | EC2/VM MariaDB only (no managed MariaDB) |
| App VM | EC2 | Compute Engine |
| Private networking | VPC + Security Groups | VPC + Firewall Rules (tag-based) |
| Static IP | Elastic IP | Static External Address |
| MariaDB port access | Security Group inbound rule | Firewall rule with target-tags |
| Redis port access | Security Group inbound rule | VPC peering (automatic for Memorystore) |

## Section 11 — Troubleshooting

### ✘ `bench new-site` hangs at MariaDB prompt

```
Cause: MariaDB not reachable — connection attempt times out
Fix:   Check security group / firewall allows port 3306
       Verify bind-address = 0.0.0.0 on MariaDB VM
       Test: mysql -h <mariadb-ip> -u root -p -e "SELECT 1;"
```

### ✘ `install-app` fails — Redis connection refused

```
Cause: Redis not reachable at configured address
       common_site_config.json still pointing to 127.0.0.1
Fix:   Update common_site_config.json with managed Redis IP
       Test: redis-cli -h <redis-ip> -p 6379 ping
```

### ✘ `bench migrate` fails — Access denied for DB user

```
Cause: Site DB user (hash user) created with host='localhost'
       App VM connects from different IP → host mismatch
Fix:   bench new-site was run without --mariadb-user-host-login-scope='%'
       Drop site and recreate with the flag
```

### ✘ Site loads but no CSS/JS (403 on assets)

```
Cause: Nginx www-data cannot traverse /home/frappe
Fix:
  chmod o+x /home/frappe
  chmod o+x /home/frappe/frappe-bench
  chmod -R o+r /home/frappe/frappe-bench/sites/assets
  sudo systemctl reload nginx
```

### ✘ Background jobs not running (emails stuck, reports not generating)

```
Cause: Redis Queue not reachable or Supervisor workers not running
Fix:
  redis-cli -h <redis-ip> -p 6379 ping          # Redis check
  sudo supervisorctl status | grep worker        # Worker check
  tail -f ~/frappe-bench/logs/worker.error.log   # Error details
```

### ✘ `bench drop-site` hangs

```
Cause: Gunicorn/worker connections holding DB metadata lock
Fix:   Stop Supervisor before dropping site
  sudo supervisorctl stop all
  bench drop-site <site> --force --no-backup
  sudo supervisorctl start all
```
