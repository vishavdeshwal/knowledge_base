# ERPNext v16 — Multi-Bench Production Setup Runbook

> **Scope:** Installing ERPNext v16 as an isolated bench on a server that already
> runs other ERPNext instances. Every step explains *why* it exists — not just what
> to type.
>
> **Naming convention used throughout:**
> - Linux user: `<bench-user>` (e.g. `miraggio`)
> - Bench directory: `/home/<bench-user>/frappe-bench`
> - Site identifier: `<site-name>` (e.g. `miraggio`)
> - Domain: `erp.<client-domain>.com`

---

## Pre-flight — Port Allocation

Every bench on the server must have **unique ports** for all five services.
If any port collides with another bench, that process silently fails or crashes
the entire `bench start`.

Check which ports are already taken before you allocate new ones:

```bash
# All gunicorn (web) ports
ss -tlnp | grep -E '800[0-9]' | sort

# All Redis ports
ss -tlnp | grep -E '1[13][0-9]{3}' | sort

# All socketio ports
ss -tlnp | grep -E '900[0-9]' | sort
```

Then assign the next available slot using this table:

```
Bench #  │ Web    │ Redis Cache │ Redis Queue │ Socketio
─────────┼────────┼─────────────┼─────────────┼─────────
  1      │  8000  │    13000    │    11000    │   9000
  2      │  8001  │    13001    │    11001    │   9001
  3      │  8002  │    13002    │    11002    │   9002
  N      │  800N  │    1300N    │    1100N    │   900N
```

> **Why five separate ports?**
> Each bench runs its own Redis processes (one for cache, one for queue) plus
> gunicorn plus socketio — all as separate OS processes. One Redis process cannot
> listen on multiple ports. Shared ports = bind conflict = BACKOFF crash loop.

## Step 0 — Verify Server-Level Prerequisites

These are installed **once on the server**, not per bench. Skip if already present.

```bash
# Python 3.14 — ERPNext v16 requires this exactly
python3.14 --version
# If missing:
sudo add-apt-repository ppa:deadsnakes/ppa -y
sudo apt update
sudo apt install -y python3.14 python3.14-dev python3.14-venv
```

### Node.js — Install via nvm (safe for multi-bench servers)

Do NOT use `sudo apt remove nodejs` — this breaks every other bench on the
server that depends on a different Node version. Use nvm instead: it installs
Node per Linux user so each bench user has their own independent Node version.

```bash
# As the bench user (not root)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash

# Load nvm into current session
source ~/.bashrc

nvm install 24
nvm use 24
nvm alias default 24   # makes v24 default for this user permanently

node --version   # must show v24.x.x
npm install -g yarn
yarn --version
```

> **Why nvm instead of apt?**
> System Node (apt) is global — removing it breaks other benches running Node 20.
> nvm puts Node in `~/.nvm/versions/node/v24.x.x/bin/node` — only visible to
> this user via PATH. Other bench users keep their own version untouched.
> When supervisor starts `node-socketio` as `<bench-user>`, it inherits this
> user's PATH and automatically picks up v24.

### Verify supervisor uses the nvm node path (after Step 10)

```bash
which node
# e.g. /home/<bench-user>/.nvm/versions/node/v24.15.0/bin/node

# After bench setup production, confirm supervisor config uses this path
grep 'node' ~/frappe-bench/config/supervisor.conf
# Must show the full nvm path, NOT /usr/bin/node

# If it shows /usr/bin/node, fix it:
NODE_PATH=$(which node)
sed -i "s|/usr/bin/node|${NODE_PATH}|g" ~/frappe-bench/config/supervisor.conf
sudo supervisorctl reread && sudo supervisorctl update
sudo supervisorctl restart <bench-user>-bench-web:
```

### Other system packages

```bash
sudo apt install -y pkg-config libmariadb-dev libmariadb-dev-compat
# pkg-config      → helps Python build system find C libraries
# libmariadb-dev  → C headers needed to compile mysqlclient Python package
#                   Without this: bench init fails with "Can not find valid pkg-config name"

# Redis memory overcommit — prevents background save failures under load
sudo sysctl vm.overcommit_memory=1
echo "vm.overcommit_memory = 1" | sudo tee -a /etc/sysctl.conf

# Ansible — pre-install to avoid PEP 668 failure during bench setup production
sudo pip3 install ansible --break-system-packages
```

> **Why Python 3.14 specifically?**
> ERPNext v16.13+ uses `requires-python = ">=3.14"` in pyproject.toml because
> Frappe v16 uses the `type` statement for type aliases — syntax only stable in 3.14.
> Python 3.12 or 3.13 hard-fail with `No solution found when resolving dependencies`.
> This is intentional, not a packaging bug.

> **Why Node 24?**
> Frappe v16 `package.json` explicitly enforces `"node": ">=24"`. Node 20 causes
> `The engine "node" is incompatible with this module` during yarn install.

## Step 1 — Create Linux User

```bash
sudo adduser <bench-user>
sudo usermod -aG sudo <bench-user>
```

> **Why a dedicated user per bench?**
> Frappe workers, gunicorn, and Redis processes all run as this user.
> If ERPNext is compromised, the attacker is limited to this user's permissions —

`OSError: Read crontab <bench-user>: fopen: Permission denied`

```bash
sudo chown root:crontab /var/spool/cron/crontabs/
```

Switch to bench user — **all remaining steps until Step 9 run as this user:**

```bash
sudo su - <bench-user>

```

## Step 4 — Fix Port Configuration

Four files must all agree on the same ports. bench init generates default
ports (13000, 11000) that are almost certainly taken by the first bench.

```bash
# 4a — Fix Redis config files (the SERVER listen ports)
grep '^port' config/redis_cache.conf config/redis_queue.conf
sed -i 's/^port .*/port 130XX/' config/redis_cache.conf
sed -i 's/^port .*/port 110XX/' config/redis_queue.conf

# 4b — Fix common_site_config.json (the CLIENT connect ports)
bench config set-common-config \
  -c redis_cache    '"redis://127.0.0.1:130XX"' \
  -c redis_queue    '"redis://127.0.0.1:110XX"' \
  -c socketio_port  90XX \
  -c webserver_port 80XX

# 4c — Fix Procfile (gunicorn port for bench start / dev mode)
sed -i 's/bench serve --port [0-9]*/bench serve --port 80XX/' ~/frappe-bench/Procfile

# 4d — Verify all four files agree
echo "=== Redis config files ===" && grep '^port' config/redis_cache.conf config/redis_queue.conf
echo "=== common_site_config ===" && cat sites/common_site_config.json | grep -E 'redis|port|socketio'
echo "=== Procfile ===" && grep web ~/frappe-bench/Procfile
```

> **Why four separate files?**
>
> `redis_cache.conf` / `redis_queue.conf` → tell Redis SERVER which port to listen on.
> Redis reads these files directly at startup.
>
> `common_site_config.json` → tells Frappe (Python CLIENT) which ports to connect to.
> Also tells socketio which port to bind. Also read by nginx config generation.
>
> `Procfile` → used only by `bench start` (dev mode honcho process).
> Completely ignored in production supervisor setup.
>
> If redis_cache.conf says 13000 but common_site_config says 13006 →
> Frappe connects to 13006 → connection refused → every page load fails.

## Step 10 — Production Setup

### What bench setup production does (and what to discard)

```
sudo bench setup production <bench-user>
  │
  ├── generates ~/frappe-bench/config/supervisor.conf   ← KEEP
  │
  ├── generates ~/frappe-bench/config/nginx.conf        ← DISCARD
  │     Always named "frappe-bench.conf" in conf.d
  │     Same filename every bench → overwrites previous bench config
  │     Use your own sites-available file instead
  │
  └── symlinks both, runs supervisorctl reread + nginx reload
```

### 10a — Run production setup

```bash
sudo bench setup production <bench-user>
```

> The argument is your **Linux username**, not the word `frappe`.
> `getpwnam(): name not found: 'frappe'` means you used the wrong argument.

### 10b — Remove auto-generated nginx config immediately

```bash
sudo rm -f /etc/nginx/conf.d/frappe-bench.conf
```

> If left in place, this conflicts with your custom sites-available config.
> nginx warns "conflicting server name... ignored" and routes traffic to the
> wrong bench. Remove it before touching nginx again.

### 10c — Fix supervisor config — rename ALL identifiers (critical step)

**Why this step exists — the supervisord flat namespace:**

supervisord is one daemon managing ALL benches on the server. It loads every
`.conf` file from `/etc/supervisor/conf.d/` into a single flat namespace.
If two conf files define the same group or program name, supervisord silently
drops the duplicate. Your bench never starts — no error, just absent from
`supervisorctl status`.

bench always generates the same generic names regardless of which user runs it:

```
frappe-bench.conf     → [group:frappe-bench-web]
miraggio-bench.conf   → [group:frappe-bench-web]   ← DUPLICATE → SILENTLY DROPPED
atb-erp-bench.conf    → [group:frappe-bench-web]   ← DUPLICATE → SILENTLY DROPPED
```

After renaming, every bench has a unique identity in supervisor's namespace:

```
frappe-bench.conf     → [group:frappe-bench-web]       ✓
miraggio-bench.conf   → [group:miraggio-bench-web]     ✓
atb-erp-bench.conf    → [group:atb-erp-bench-web]      ✓
```

**Three line types that all need renaming:**

supervisor.conf has three types of lines that contain `frappe-bench`:

```ini
[program:frappe-bench-frappe-web]      ← defines individual process
[group:frappe-bench-web]               ← defines a group of processes
programs=frappe-bench-frappe-web,...   ← links programs into their group
```

If you rename `[program:]` and `[group:]` but miss `programs=`,
supervisor finds the group but can't locate the programs inside it:

```ini
[group:miraggio-bench-redis]              ← group renamed ✓
programs=frappe-bench-redis-cache,...     ← still old name ✗
→ supervisor: cannot find "frappe-bench-redis-cache"
→ result: ERROR (spawn error) on every process in this group
```

**The correct single sed — targets all three line types, never touches paths:**

```bash
sed -i '/^\[program:\|^\[group:\|^programs=/ s/frappe-bench/<bench-user>-bench/g' \
  ~/frappe-bench/config/supervisor.conf
```

Breaking this down:
- `/^\[program:\|^\[group:\|^programs=/` — line filter: only applies substitution
  to lines starting with `[program:`, `[group:`, or `programs=`
- `s/frappe-bench/<bench-user>-bench/g` — renames within those lines only
- Lines like `directory=`, `command=`, `stdout_logfile=` start differently
  → untouched → filesystem paths stay correct

**Verify all three types renamed, paths untouched:**

```bash
grep -E '^\[program:|^\[group:|^programs=' \
  ~/frappe-bench/config/supervisor.conf
# Must show <bench-user>-bench everywhere — zero occurrences of frappe-bench

grep 'directory=' ~/frappe-bench/config/supervisor.conf
# Must show /home/<bench-user>/frappe-bench — NOT <bench-user>-bench
```

Expected output:
```ini
[program:<bench-user>-bench-frappe-web]         ✓
[program:<bench-user>-bench-redis-cache]        ✓
[program:<bench-user>-bench-redis-queue]        ✓
[group:<bench-user>-bench-web]                  ✓
programs=<bench-user>-bench-frappe-web,...      ✓
[group:<bench-user>-bench-redis]                ✓
programs=<bench-user>-bench-redis-cache,...     ✓
[group:<bench-user>-bench-workers]              ✓
programs=<bench-user>-bench-frappe-schedule,... ✓

directory=/home/<bench-user>/frappe-bench       ✓ path untouched
```


## Step 11 — nginx Configuration

Do not use `bench setup nginx` or the auto-generated `conf.d/frappe-bench.conf`.
Write your own file in `sites-available` named after your domain.

**Why sites-available + sites-enabled vs conf.d:**

```
conf.d/                               sites-available/ + sites-enabled/
──────────────────────────────────    ──────────────────────────────────
Auto-loaded, all files always active  sites-available = stored configs
bench always writes frappe-bench.conf One file per domain, named by domain
Same filename → overwrites previous   No filename conflicts ever
No enable/disable mechanism           Disable: rm sites-enabled/<domain>
Certbot may edit wrong file           Certbot edits exactly this file
```

### Write the nginx config

```bash
sudo vi /etc/nginx/sites-available/erp.<client-domain>.com
```

```nginx
upstream <bench-user>-bench {
    server 127.0.0.1:80XX fail_timeout=0;
}

upstream <bench-user>-socketio {
    server 127.0.0.1:90XX fail_timeout=0;
}

server {
    listen 80;
    listen [::]:80;
    server_name erp.<client-domain>.com;

    root /home/<bench-user>/frappe-bench/sites;
    client_max_body_size 50m;

    proxy_buffer_size 128k;
    proxy_buffers 4 256k;
    proxy_busy_buffers_size 256k;

    add_header X-Frame-Options "SAMEORIGIN";
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload";
    add_header X-Content-Type-Options nosniff;
    add_header X-XSS-Protection "1; mode=block";
    add_header Referrer-Policy "same-origin, strict-origin-when-cross-origin";

    # Static assets — served directly by nginx, never hits Python
    location /assets {
        try_files $uri =404;
        add_header Cache-Control "max-age=31536000";
    }

    location ~ ^/protected/(.*) {
        internal;
        try_files /<site-name>/$1 =404;
    }

    # WebSocket → socketio for realtime updates, progress bars, notifications
    location /socket.io {
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header X-Frappe-Site-Name <site-name>;
        proxy_set_header Origin $scheme://$http_host;
        proxy_set_header Host $host;
        proxy_pass http://<bench-user>-socketio;
    }

    location / {
        rewrite ^(.+)/$ $1 permanent;
        rewrite ^(.+)/index\.html$ $1 permanent;
        rewrite ^(.+)\.html$ $1 permanent;

        location ~* ^/files/.*\.(htm|html|svg|xml) {
            add_header Content-disposition "attachment";
            try_files /<site-name>/public/$uri @webserver;
        }

        try_files /<site-name>/public/$uri @webserver;
    }

    # All dynamic requests → gunicorn on 80XX
    location @webserver {
        proxy_http_version 1.1;
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Frappe-Site-Name <site-name>;
        proxy_set_header Host $host;
        proxy_set_header X-Use-X-Accel-Redirect True;
        proxy_read_timeout 120;
        proxy_redirect off;
        proxy_pass http://<bench-user>-bench;
    }

    error_page 502 /502.html;
    location /502.html {
        root /usr/local/lib/python3.12/dist-packages/bench/config/templates;
        internal;
    }

    sendfile on;
    keepalive_timeout 15;
    client_body_buffer_size 16K;
    client_header_buffer_size 1k;

    gzip on;
    gzip_http_version 1.1;
    gzip_comp_level 5;
    gzip_min_length 256;
    gzip_proxied any;
    gzip_vary on;
    gzip_types
        application/atom+xml application/javascript application/json
        application/rss+xml application/vnd.ms-fontobject
        application/x-font-ttf application/font-woff
        application/xhtml+xml application/xml font/opentype
        image/svg+xml image/x-icon text/css text/plain text/x-component;
}
```

> **How nginx tells Frappe which site to serve:**
> The header `proxy_set_header X-Frappe-Site-Name <site-name>` is passed to
> gunicorn on every request. Frappe reads this header and loads the correct
> site's database credentials from `sites/<site-name>/site_config.json`.
> Without this header, Frappe falls back to the Host header — if that doesn't
> exactly match a site name, you get a 404 "does not exist" error.

### Enable and reload

```bash
sudo ln -sf /etc/nginx/sites-available/erp.<client-domain>.com \
  /etc/nginx/sites-enabled/erp.<client-domain>.com

sudo nginx -t && sudo systemctl reload nginx
```

### Fix www-data traversal permissions (always required)

```bash
chmod o+x /home/<bench-user>
chmod o+x /home/<bench-user>/frappe-bench
chmod -R o+r /home/<bench-user>/frappe-bench/sites/assets

# Verify nginx can reach assets
sudo -u www-data ls /home/<bench-user>/frappe-bench/sites/assets/
# Must list files — Permission denied means chmod above didn't work
```

> nginx runs as `www-data`. Without execute on the home directory path,
> www-data cannot traverse to `sites/assets/` — all CSS/JS returns 403
> and the site loads as unstyled plain HTML.

## How supervisorctl Differentiates Multiple Benches — Full Mental Model

```
/etc/supervisor/conf.d/              supervisord flat namespace (merged)
──────────────────────────────────   ──────────────────────────────────────
atb-erp-frappe-bench.conf            [program:atb-erp-bench-frappe-web]
  symlink →                          [program:atb-erp-bench-redis-cache]
  /home/atb-erp/frappe-bench/        [group:atb-erp-bench-web]
  config/supervisor.conf               programs=atb-erp-bench-frappe-web,...
                                     [group:atb-erp-bench-redis]
                                       programs=atb-erp-bench-redis-cache,...

miraggio-bench.conf                  [program:miraggio-bench-frappe-web]
  symlink →                          [program:miraggio-bench-redis-cache]
  /home/miraggio/frappe-bench/       [group:miraggio-bench-web]
  config/supervisor.conf               programs=miraggio-bench-frappe-web,...
                                     [group:miraggio-bench-redis]
                                       programs=miraggio-bench-redis-cache,...
```

Every name in this namespace must be unique. supervisorctl commands reference
these names directly:

```bash
sudo supervisorctl status | grep atb-erp-bench     # only atb-erp processes
sudo supervisorctl restart miraggio-bench-web:      # only miraggio web group
sudo supervisorctl stop atb-erp-bench-workers:      # only atb-erp workers
```

If names are not unique, the wrong bench's process may respond to your command,
or the command fails silently because the group was never loaded.

## Operational Reference

### Restart commands

```bash
# After code change / install-app / update (reloads Python module state)
sudo supervisorctl restart <bench-user>-bench-web:
sudo supervisorctl restart <bench-user>-bench-workers:

# After only a Python/backend change (fastest — just gunicorn)
sudo supervisorctl restart <bench-user>-bench-web:<bench-user>-bench-frappe-web

# Nuclear option — avoid (drops Redis cache and all queued jobs)
sudo supervisorctl restart <bench-user>-bench-web: \
  <bench-user>-bench-workers: \
  <bench-user>-bench-redis:
```

> **Why restart after install-app?**
> Gunicorn workers load all Python modules into memory at startup. Installing
> a new app adds files to disk but running workers keep old import state.
> `import new_app` → ModuleNotFoundError → 500 on every page.
> Restarting forces fresh imports.

### Deploy sequence (every code change)

```bash
cd ~/frappe-bench
bench get-app --branch version-16 <app>         # if new app
bench --site <site-name> install-app <app>       # if new app
```

## Do's and Don'ts

### DO

- Always check `grep '^port' config/redis_cache.conf config/redis_queue.conf` before `bench start`
- Always pass `--python python3.14` to `bench init`
- Always use nvm for Node.js — never remove system Node with apt
- Always use the three-line-type sed:
  `sed -i '/^\[program:\|^\[group:\|^programs=/ s/frappe-bench/<bench-user>-bench/g'`
- Always verify `programs=` lines renamed — missing this causes spawn error
- Always verify paths after sed: `grep 'directory=' ~/frappe-bench/config/supervisor.conf`
- Always remove `conf.d/frappe-bench.conf` after `bench setup production`
- Always write nginx in `sites-available/<domain>` with domain as filename
- Always run `sudo nginx -t` before `systemctl reload nginx`
- Always restart gunicorn after `install-app`, `migrate`, or any code change
- Always run `bench --site <name> clear-cache` after any change
- Always start Redis before gunicorn when starting process groups

### DON'T

- Don't run `bench setup production frappe` — use your actual Linux username
- Don't use `bench start` in production
- Don't use `bench setup nginx` on a multi-bench server
- Don't run `bench setup nginx` after SSL — it wipes certbot SSL additions
- Don't run global sed without line filter — it breaks filesystem paths
- Don't forget `programs=` lines — only renaming `[program:]` and `[group:]` causes spawn error
- Don't share Redis between benches — each bench needs its own processes
- Don't restart Redis unnecessarily — drops cached sessions and queued jobs
- Don't assume login failure is a password issue — first confirm nginx routes to correct bench/port
- Don't give bench users full sudo — use scoped sudoers

## Architecture Reference

```
ONE nginx (root)
  ├── sites-enabled/erp.<client1>.com  → upstream :8006 (bench 1)
  ├── sites-enabled/erp.<client2>.com  → upstream :8007 (bench 2)
  └── sites-enabled/erp.<client3>.com  → upstream :8008 (bench 3)
  Routes by Host header (server_name). Each file unique. No conflicts.

ONE supervisord (root) — flat namespace, all names must be unique
  ├── conf.d/atb-erp-frappe-bench.conf → symlink → /home/atb-erp/.../supervisor.conf
  │     [group:atb-erp-bench-web]       programs=atb-erp-bench-frappe-web,...
  │     [group:atb-erp-bench-redis]     programs=atb-erp-bench-redis-cache,...
  │     all processes run AS atb-erp (user= directive in each [program:])
  │
  ├── conf.d/miraggio-bench.conf        → symlink → /home/miraggio/.../supervisor.conf
  │     [group:miraggio-bench-web]      programs=miraggio-bench-frappe-web,...
  │     [group:miraggio-bench-redis]    programs=miraggio-bench-redis-cache,...
  │     all processes run AS miraggio
  └── ...

ONE MariaDB — separate database per site, separate DB user per bench

PER-BENCH processes (run as bench user):
  gunicorn   :80XX  → Python WSGI — handles all HTTP requests
  socketio   :90XX  → Node.js — handles WebSocket realtime updates
  redis      :130XX → cache (sessions, permissions, boot data)
  redis      :110XX → queue (emails, reports, background jobs)
  workers    (no port) → consume jobs from redis queue asynchronously
  scheduler  (no port) → fires recurring cron-style jobs
```
