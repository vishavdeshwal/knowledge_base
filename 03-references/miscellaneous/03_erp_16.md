# ERPNext Version 16 — Setup Guide

## What Is ERPNext?

**ERPNext** is a free, open-source, full-featured **Enterprise Resource Planning (ERP)** system. ERP means one unified system that runs all business operations — instead of separate tools for accounting, HR, inventory, and sales that don't talk to each other, ERPNext puts everything in one database with one interface.

ERPNext handles: Accounting & Finance, HR & Payroll, Inventory & Warehouse, CRM & Sales, Manufacturing, Project Management, Purchase & Supply Chain.

But ERPNext is not a standalone application. It is built on top of **Frappe Framework** — and you cannot run ERPNext without Frappe. To understand what you are installing and why, you need to understand this layered relationship clearly.

Think of it like this: **Frappe is WordPress, ERPNext is a WordPress theme/plugin.** The CMS engine is Frappe. The business application on top is ERPNext.

## What Is a Doctype?

Everything in Frappe/ERPNext is a **Doctype**. A Doctype is a blueprint — it defines a database table, a form in the UI, and the logic around it, all in one place.

```
Doctype: "Sales Invoice"
  │
  ├── Fields definition
  │     → customer (Link to Customer doctype)
  │     → items (Child table → Sales Invoice Item)
  │     → grand_total (Currency field)
  │     → status (Select: Draft/Submitted/Cancelled)
  │
  ├── Database table: `tabSales Invoice`
  │     → Frappe auto-creates this in MariaDB
  │     → Every field = one column
  │
  ├── REST API: auto-generated by Frappe
  │     GET  /api/resource/Sales Invoice/<name>
  │     POST /api/resource/Sales Invoice
  │
  └── Form UI: auto-generated by Frappe
        → JS renders the form from field metadata
        → No manual HTML needed
```

This means when ERPNext ships a new doctype — say "Salary Component" — Frappe automatically creates the table in MariaDB, generates the REST API, and renders the form. ERPNext only writes the business logic on top.

## What Each Component Does — Micro Detail

### Nginx
The first thing a browser request hits. Nginx does three things:
1. **Terminates SSL** — decrypts HTTPS, passes plain HTTP internally
2. **Serves static files directly** — CSS, JS, images from `sites/assets/` never reach Python. Nginx handles them alone, which is why file permissions on `/home/frappe` matter
3. **Reverse proxies everything else** → forwards to Gunicorn on `127.0.0.1:8000`

```
Browser → Nginx (443) → static? serve directly
                      → dynamic? proxy_pass → Gunicorn (8000)
```

---

### Gunicorn
A Python WSGI server. It is the process that actually runs Frappe's Python code. When Nginx forwards a request, Gunicorn picks it up and hands it to Frappe's request handler.

```
Gunicorn spawns N worker processes (configurable via gunicorn_workers)
Each worker = one Python process that can handle one request at a time
5 workers = 5 simultaneous requests handled in parallel
```

---

### Frappe Framework (inside Gunicorn)
Once Gunicorn hands the request to Frappe:

```
Request arrives
      │
      ▼
1. Parse Host header → identify which site (multi-tenancy)
      │
      ▼
2. Load site_config.json → get DB credentials for this site
      │
      ▼
3. Authenticate → check session token in Redis Cache
      │
      ▼
4. Route to controller → e.g. /api/resource/Sales Invoice
      │
      ▼
5. ORM executes SQL → SELECT * FROM `tabSales Invoice` WHERE ...
      │
      ▼
6. Run hooks → before_save, validate, on_submit (ERPNext logic runs here)
      │
      ▼
7. Return JSON response → Gunicorn → Nginx → Browser
```

---

### MariaDB
Stores every single piece of business data. Each Frappe Doctype becomes a table prefixed with `tab`:

```
Doctype              MariaDB Table
────────────────────────────────────
Sales Invoice     →  tabSales Invoice
Customer          →  tabCustomer
Item              →  tabItem
User              →  tabUser
GL Entry          →  tabGL Entry  (every accounting transaction)
```

Frappe's ORM translates Python code like `frappe.get_doc("Sales Invoice", "SINV-0001")` into `SELECT * FROM \`tabSales Invoice\` WHERE name = 'SINV-0001'` automatically.

---

### Redis — Three Roles

Redis is used for three completely separate purposes in Frappe, each on its own port:

```
Redis Cache (port 13000)
  └── Stores: user sessions, permission cache, boot data, metadata
      Why: MariaDB is slow for repeated reads of the same data.
           Redis answers in microseconds vs milliseconds.
      If Redis Cache goes down: every page load hits MariaDB directly → very slow

Redis Queue (port 11000)
  └── Stores: background job payloads (send email, generate report, import CSV)
      Why: You don't want the HTTP request to wait while an email sends.
           Frappe pushes the job to Redis Queue and returns immediately.
           A separate Worker process picks it up asynchronously.
      If Redis Queue goes down: all background jobs stop silently

Redis SocketIO (shared with Cache in v15)
  └── Stores: real-time pub/sub messages
      Why: When someone saves a document, other users' browsers
           get notified in real-time via WebSocket.
      If this goes down: no real-time updates, progress bars freeze
```

---

### Frappe Worker (Background Process)
A separate Python process that runs independently of Gunicorn. It watches Redis Queue, picks up jobs, and executes them.

```
User clicks "Send Email" in ERPNext
      │
      ▼
Frappe pushes job to Redis Queue → returns "queued" to browser instantly
      │
      ▼ (asynchronously, seconds later)
Frappe Worker picks job from Redis Queue
      │
      ▼
Worker executes: connects to MariaDB, reads email template,
                 sends via SMTP, updates Email Queue doctype
```

---

### Frappe Scheduler
Another separate background process. Runs recurring tasks on a cron-like schedule.

```
Every minute  → check for scheduled jobs in tabScheduled Job Log
Every hour    → run hourly hooks (e.g. update currency exchange rates)
Every day     → run daily hooks (e.g. auto-create recurring invoices)
Every week    → run weekly hooks
```

Without Scheduler running, things like auto-reminders, recurring invoices, and scheduled reports never fire.

---

### Node.js + yarn (Build-time only)
Node.js is **not a runtime dependency** once the server is running. It is only used during:
1. `bench build` — compiles Vue.js components + LESS/SCSS into minified CSS/JS bundles
2. `bench get-app` / `bench update` — installs JS dependencies via yarn

```
Source files (apps/erpnext/public/js/*.js)
      │
      ▼  yarn + webpack (via bench build)
      │
Compiled bundles (sites/assets/erpnext/js/erpnext.min.js)
      │
      ▼
Nginx serves these directly to the browser
```

If Node.js is missing or wrong version → `bench build` fails → browser gets old/broken JS.

---

### Supervisor
Supervisor is the process manager that keeps all Frappe processes alive on the server. Without it, if Gunicorn crashes or the server reboots, nothing restarts automatically.

```
Supervisor manages:
  ├── frappe-web
  │     └── Gunicorn (the Python web server)
  ├── frappe-node-socketio
  │     └── Node SocketIO server (real-time)
  ├── frappe-redis-cache
  │     └── Redis instance on port 13000
  ├── frappe-redis-queue
  │     └── Redis instance on port 11000
  ├── frappe-schedule
  │     └── Frappe Scheduler process
  ├── frappe-short-worker
  │     └── Worker for short jobs (emails, notifications)
  └── frappe-long-worker
        └── Worker for long jobs (bulk imports, report generation)
```

If Supervisor config is not symlinked to `/etc/supervisor/conf.d/`, none of these processes start — which is exactly the silent failure encountered in this setup.

---

### bench CLI
bench is not a server process — it is a developer/admin tool. It:
- Initialises a new bench environment (`bench init`)
- Downloads apps from GitHub (`bench get-app`)
- Creates sites and databases (`bench new-site`)
- Installs apps onto sites (`bench --site x install-app y`)
- Builds JS assets (`bench build`)
- Generates Nginx and Supervisor configs (`bench setup production`)
- Runs database migrations after updates (`bench migrate`)

```
bench
  └── reads: ~/frappe-bench/sites/currentsite.txt  (default site)
  └── writes: ~/frappe-bench/config/nginx.conf
              ~/frappe-bench/config/supervisor.conf
  └── executes: python, yarn, pip inside ~/frappe-bench/env/
```

## How a Full Browser Request Flows — End to End

```
1. User opens https://erp-staging.beyondage.infinitelocus.com/app/sales-invoice

2. DNS resolves domain → server IP

3. Nginx receives request on port 443
   └── Decrypts SSL
   └── Checks: is this /assets/ or /files/? No → forward to Gunicorn

4. Gunicorn receives on port 8000
   └── Hands to Frappe Python app

5. Frappe reads Host header: erp-staging.beyondage.infinitelocus.com
   └── Matches to site: beyondage (via domains[] in site_config.json)
   └── Loads DB creds from sites/beyondage/site_config.json

6. Frappe checks Redis Cache for session token
   └── Valid session found → user is authenticated

7. Frappe routes /app/sales-invoice → desk (SPA shell)
   └── Returns HTML shell + links to JS/CSS bundles

8. Browser fetches JS/CSS from /assets/
   └── Nginx serves directly from sites/assets/ (no Python involved)

9. Vue.js app boots in browser
   └── Makes API call: GET /api/resource/Sales Invoice?limit=20

10. Frappe ORM: SELECT * FROM `tabSales Invoice` LIMIT 20
    └── MariaDB returns rows
    └── Frappe serialises to JSON
    └── Response flows back: MariaDB → Frappe → Gunicorn → Nginx → Browser

11. User sees Sales Invoice list rendered in Vue.js
```

# ERPNext Version 16 — Setup Guide
 
## What's Different in v16
 
Frappe v16 is a major release with significantly stricter system requirements compared to v15. The core runtime stack changed substantially — Python, Node.js, and several build dependencies have new minimum versions. **v16 is not backward compatible with the v15 setup process.**
 
---
 
## v16 vs v15 — Requirements Comparison
 
| Component | v15 | v16 | Notes |
|---|---|---|---|
| Python | 3.10 / 3.11 | **3.14** | v16 uses `type` alias syntax — Python 3.12/3.13 not sufficient |
| Node.js | 20 | **24** | Frappe v16 `package.json` enforces `engine >= 24` |
| MariaDB | 10.6+ | 10.6+ | Same — local MariaDB works fine |
| MySQL | 8.0 | 8.0 only | MySQL 8.4 incompatible — see Cloud SQL section |
| Redis | 6+ | 6+ | Same |
| pkg-config | Not needed | **Required** | New Python deps with C extensions need it |
| libmariadb-dev | Not needed | **Required** | mysqlclient compilation requires dev headers |
| `--no-mariadb-socket` | Works | **Deprecated** | Use `--mariadb-user-host-login-scope='%'` |

## Step 1 — System Packages (v16 additions)
 
**What changes:** Two new packages required — `pkg-config` and `libmariadb-dev`. Without these, `bench init` fails when compiling `mysqlclient`.
 
```bash
sudo apt update && sudo apt upgrade -y
 
echo '* libraries/restart-without-asking boolean true' | sudo debconf-set-selections

  git curl wget \
  python3 python3-dev python3-pip python3-venv \
  build-essential \
  libffi-dev libssl-dev \
  software-properties-common \
  xvfb libfontconfig \
  cron \

  pkg-config \
  libmariadb-dev \
  libmariadb-dev-compat
 
# Verify pkg-config can find MariaDB headers
pkg-config --exists mariadb && echo "mariadb headers found ✔" || echo "not found ✗"
```
 
> `pkg-config` — helps Python's build system locate installed C libraries during compilation.
> `libmariadb-dev` — MariaDB C development headers needed by `mysqlclient` Python package.
> Without `libmariadb-dev`, bench init fails with: `Can not find valid pkg-config name`.

## Step 2 — Python 3.14 (NEW for v16)
 
**What this does:** Installs Python 3.14 via the deadsnakes PPA. Ubuntu 22.04 ships Python 3.10 which is incompatible with Frappe v16.
 
> ⚠️ **Why 3.14 specifically?** Frappe v16 uses the `type` statement for type aliases:
> ```python
> type ConfType = _dict[str, Any]  # valid Python 3.12+ syntax
> ```
> Additionally, Frappe v16.13+ bumped its requirement to `Python>=3.14,<3.15`.
> Python 3.12 and 3.13 are not sufficient.
 
```bash

sudo apt update
 
# Check 3.14 is available

# Install
sudo apt install -y python3.14 python3.14-dev python3.14-venv

```

### Secure MariaDB and Switch Auth Plugin
 
> ⚠️ **CRITICAL — Answer N to unix_socket prompt.**
> During `mysql_secure_installation`, answer **N** to "Switch to unix_socket authentication".
> If you answer Y, bench will fail with `(1698) Access denied` at `bench new-site`.
> Run the `ALTER USER` below regardless of your answer — it guarantees password auth.
 
```bash
sudo mysql_secure_installation
# Switch to unix_socket authentication? → N
# Change root password? → Y (set a strong password)
# Remove anonymous users? → Y
# Disallow remote root login? → Y
# Remove test database? → Y
# Reload privilege tables? → Y
```
 
Switch root to password auth immediately after:
 
```bash
sudo mysql
```
 
```sql
ALTER USER 'root'@'localhost'
  IDENTIFIED VIA mysql_native_password
  USING PASSWORD('<db-root-password>');
FLUSH PRIVILEGES;
EXIT;
```
 
### Configure MariaDB Charset
 
> ⚠️ **CRITICAL — Section order `[server] → [mysql] → [mysqld]` is mandatory.**
> Placing `[mysqld]` above `[mysql]` causes collation errors at `bench new-site`.
 
```bash
sudo nano /etc/mysql/mariadb.conf.d/99-frappe.cnf
```
 
```ini
[server]
 
[mysql]
default-character-set = utf8mb4
 
[mysqld]
character-set-client-handshake = FALSE
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
```
 
```bash
sudo systemctl restart mariadb
 
# Verify — without sudo (bench runs as frappe user)
mysql -u root -p'<db-root-password>' -e "SHOW VARIABLES LIKE 'character_set_server';"
# Must return: utf8mb4
```

## Step 8 — Install bench CLI
 
**What this does:** Installs the `bench` CLI tool globally so it can be used to initialise bench environments, manage sites, and run apps.
 
> ⚠️ **PEP 668 — Python 3.12+ blocks system-wide pip installs.**
> Ubuntu 22.04 with Python 3.12 installed marks the system Python as "externally managed". Running `sudo pip3 install` without a flag will fail with `externally-managed-environment`.
> This is a Python 3.12+ protection mechanism — not present in Python 3.10.
> On v16 setups Python 3.12 is often pre-installed by the Ubuntu image alongside 3.14.
 
Check which Python `pip3` points to:
```bash
pip3 --version
# If it shows python 3.12 or higher → use --break-system-packages
# If it shows python 3.10 → standard install works fine
```
 
### Option 1 — pip with `--break-system-packages` (recommended for v16)
 
```bash
sudo pip3 install frappe-bench --break-system-packages
bench --version
```
 
> `--break-system-packages` is safe for `frappe-bench` — it is a standalone CLI tool, not a system dependency. The flag bypasses PEP 668 protection for this specific install.
 
### Option 2 — pipx (cleanest approach)
 
```bash
sudo apt install -y pipx
pipx install frappe-bench
pipx ensurepath
source ~/.bashrc
 
bench --version
```
 
> `pipx` installs CLI tools in isolated virtual environments — exactly what PEP 668 recommends. No system Python is modified.
 
---
 
### Step 8a — Pre-install System Dependencies for bench (v16 only)
 
**Why this step exists:** bench internally runs `sudo pip install ansible` during `bench setup production` (Step 13). On Python 3.12+ systems this fails with PEP 668. Pre-installing ansible now prevents a hard stop later.
 
> ⚠️ **Do this immediately after installing frappe-bench — before any other bench commands.**
> If skipped, `bench setup production` will fail at Step 13 with:
> ```
> error: externally-managed-environment
> ERROR: sudo /usr/bin/python3 -m pip install ansible
> ```
 
```bash
# Pre-install all system-level dependencies bench needs
sudo pip3 install ansible --break-system-packages
 
# Verify
ansible --version | head -1
# Should show: ansible [core x.x.x]
```
 
**All PEP 668 affected commands in v16 setup:**
 
| Step | Command bench runs internally | Pre-fix |
|---|---|---|
| Step 8 | `pip install frappe-bench` | `--break-system-packages` |
| Step 8a | `pip install ansible` | pre-install here ← |
| Step 13 | `bench setup production` calls ansible | blocked if not pre-installed |

```bash

```
 
**What `bench init` checks during v16 init:**

## Database Compatibility — MariaDB vs MySQL vs Cloud SQL
 
### Why Frappe Uses MariaDB as Primary Database
 
Frappe was originally built on MySQL but migrated to MariaDB as its primary target around 2013 when the MariaDB community fork gained momentum. Today MariaDB is Frappe's most tested and officially supported database engine.
 
```
Frappe Official DB Support:
  ✔ MariaDB 10.6+    ← primary, most tested, recommended
  ✔ MySQL 8.0        ← secondary, works with correct config
  ✔ PostgreSQL 13+   ← alternative path, use --db-type postgres
  ✗ MySQL 8.4        ← not in support matrix, breaking changes
```
 
---
 
### MariaDB vs MySQL — Key Differences Affecting Frappe
 
MariaDB and MySQL forked in 2009. They have diverged significantly. The differences below directly affect Frappe installation and operation:
 
| Feature | MariaDB 10.6 | MySQL 8.0 | MySQL 8.4 |
|---|---|---|---|
| `mysql_native_password` | ✅ Supported | ✅ Supported | ❌ Removed |
| MyISAM storage engine | ✅ Available | ✅ Available | ❌ Disabled by default |
| Password policy | ✅ Off by default | ✅ Configurable | ⚠️ Strict on Cloud SQL |
| Frappe support | ✅ Primary | ✅ Secondary | ❌ Not supported |
| `bench new-site` | ✅ Works | ✅ Works with flags | ❌ Fails |
 
---
 
### Why Each Difference Breaks Frappe
 
#### 1. `mysql_native_password` — Auth Plugin
 
Frappe uses `mysqlclient` — a Python C extension — to connect to the database. This driver uses password-based authentication. MariaDB and MySQL 8.0 support `mysql_native_password` which `mysqlclient` handles natively.
 
MySQL 8.4 removed this plugin entirely:
 
```
bench new-site → Frappe tries to CREATE USER with mysql_native_password
MySQL 8.4 responds → ERROR 4052: Invalid plugin "mysql_native_password"
Result → Site creation fails immediately
```
 
On MariaDB:
```
bench new-site → CREATE USER with mysql_native_password
MariaDB responds → OK ✔
Result → Site created cleanly
```
 
#### 2. MyISAM Storage Engine
 
Frappe creates a `__global_search` table using the MyISAM storage engine. MyISAM provides full-text search capabilities used across all doctypes in ERPNext.
 
```
bench new-site → CREATE TABLE __global_search ENGINE=MyISAM
MySQL 8.4 (Cloud SQL) → ERROR 3161: Storage engine MyISAM is disabled
MariaDB → OK ✔
```
 
On Cloud SQL MySQL 8.4 specifically — Google permanently disabled MyISAM. The `disabled_storage_engines` flag is not available in the Cloud SQL console flags list for 8.4. There is no workaround.
 
#### 3. Password Policy on Cloud SQL
 
When `bench new-site` creates a dedicated DB user for the site, it generates a random password internally. On Cloud SQL MySQL 8.4, Google's custom password policy (`cloudsql_password_validation_policy_enable`) rejects this random password because it doesn't meet complexity requirements.
 
```
bench new-site → generates random password like 'Zd4Mlwxq7HCjnNCf'
Cloud SQL policy → ERROR 9000: Password Policy violated
Result → DB user creation fails
```
 
This policy cannot be disabled:
- `SET GLOBAL cloudsql_password_validation_policy_enable = OFF` → unknown variable
- Cloud SQL Console → Flags → not listed as configurable flag on 8.4

### Pre-flight Checklist for Cloud SQL MySQL 8.0

```bash

### bench Redis Must Be Running During `install-app` in v16
 
**This is a new behaviour confirmed in v16.** In v15, `install-app` runs cleanly without Redis. In v16, ERPNext's `after_install` hook pushes background jobs to Redis Queue during installation. If Redis is not running, install fails with:

**Fix:** Always start bench processes before running `install-app` in v16:
 
```bash

### Do NOT Run `bench setup nginx` After Certbot in v16
 
Same as v15 — `bench setup nginx` regenerates the config from scratch and wipes certbot's SSL additions.
 
```
After certbot runs:
  /etc/nginx/conf.d/frappe-bench.conf → has listen 443 ssl + cert paths
 
After bench setup nginx:
  /etc/nginx/conf.d/frappe-bench.conf → regenerated, SSL config GONE
  Site reverts to HTTP only
```

```bash

# Then validate and reload
sudo nginx -t && sudo systemctl reload nginx
 
# If SSL was wiped — rerun certbot
sudo certbot --nginx -d erp.<client-application-name>.com
```
